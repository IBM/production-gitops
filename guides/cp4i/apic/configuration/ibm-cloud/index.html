    IBM API Connect configuration on IBM Cloud¶

    In the previous chapter of this tutorial, we went through what needs to be done within our GitOps repository in order to get a highly available and production ready IBM API Connect cluster deployed on a Red Hat OpenShift cluster on IBM Cloud. However, that deployment process, as the name indicates, only deploys the different subsystems that an IBM API Connect cluster is made of.

    Overview¶

    An IBM API Connect cluster (defined through its APIConnectCluster Custom Resource Definition - CRD), deploys the following subsystems:

    • Management Cluster (ManagementCluster CRD)
    • Portal Cluster (PortalCluster CRD)
    • Gateway Cluster (GatewayCluster CRD)
    • Analytics Cluster (AnalyticsCluster CRD)

    Tip

    The reason for the following subsystems to be called clusters is because each of those IBM API Connect subsystems can be easily deployed highly available by using the profile property. As a result, it is common to see and strongly recommended, specially on production environment, to deploy all of these IBM API Connect subsystems using their highly available profile which will deploy 3 replicas of each.

    However, as already introduced, the deployment process will not configure all these IBM API Connect subsystems to work together straight away. Instead, you will need to complete the following IBM API Connect Cloud Manager configuration checklist that will drive you through things like Register a Gateway service, Register an Analytics service, Create a provider organization, etc.

    Automation¶

    Similarly to deploying any piece of software in production, we would like to be able to configure those pieces of software through a GitOps methodology as well. However, the configuration of some of those pieces of software might not yet be enabled be done following a GitOps methodology. Still, we would like to be able to configure any of those pieces of software or components through certain degree of automation. Automation will provide us with two major benefits: auditability and repetition (both of which are inherent to GitOps).

    Since the configuration of theses pieces of software is done through automation and automation is nothing but code describing actions to be done/taken, that pieces of code that is automating the configuration of a piece of software is auditable and repeatable. These two properties allow us to see how exactly a piece of software has been configured in case we need to adjust anything and also give us that repeatability that provides us with a kind of disaster recovery mechanism that will allow us to configure another instance of that piece of software in the exact same manner as its sibling. Of course, these capabilities are a must in production environments.

    The automation we are describing and looking for to automate the IBM API Connect Cloud Manager configuration checklist can be achieved thanks to Tekton (OpenShift Pipelines) and, in the particular case of IBM API Connect, the rich set of IBM API Connect administration APIs. With these two, we have managed to create an OpenShift pipeline that will automate the following actions for use to configure a new deployment of an IBM API Connect instance:

    1. Configure an email server for notifications.
    2. Register the default Gateway Service.
    3. Register the default Analytics Service.
    4. Associate the default Analytics Service with the default Gateway Service.
    5. Register the default Portal Service.
    6. Create a Provider Organization.

    7. Associate the default Gateway Service with the default Sandbox catalog.

    Important

    The IBM API Connect configuration automation provided in this Cloud Pak Production Guides should be use as reference and never as a production ready asset that will suit every client needs. Each client will have a particular set of requirements and architecture and, as a result, the automation presented here can be use as a template or starting point but it is not mean to provide a production ready solution by any means.

    OpenShift pipeline¶

    Let's see how can we create and execute the OpenShift pipeline that will configure our recent deployment of an IBM API Connect instance on IBM Cloud.

    1. Locate your GitOps repository

      If necessary, change to the root of your GitOps repository, which is typically $HOME/git.

      Issue the following command to change to your GitOps repository:

      cd $HOME/git
cd multi-tenancy-gitops

    2. Open the 0-bootstrap/single-cluster/1-infra/kustomization.yaml file and un-comment the following resource in order to create the ci namespace where the OpenShift pipeline will get created and executed from:

      - argocd/namespace-ci.yaml

    3. Open the 0-bootstrap/single-cluster/3-apps/kustomization.yaml file and un-comment the following resource in order to create the apps-apic-ci ArgoCD application that will take care of creating the IBM API Connect configuration pipeline:

      - argocd/apic/cicd.yaml

    4. Add all changes in the current folder to a git index, commit them, and push them to GitHub:

      git add .
git commit -s -m "Creating APIC pipelines"
git push origin $GIT_BRANCH

    5. If you check your ArgoCD UI, you should now see few new ArgoCD applications as a result of the changes above:

    6. If you go to Pipelines --> Pipelines on the left hand side menu of your OpenShift web console, you will see that two new OpenShift pipelines have been created as a result of the previous changes to our GitOps repository:

      Info

      The pipeline we are interested in for getting our instance of IBM API Connect configured is called apic-post-install-config

    Before executing the OpenShift pipeline that will configure our instance of IBM API Connect, we need to provide some configuration to the actual OpenShift pipeline to be successful.

    1. Ensure you are logged in to the cluster

      Start a terminal window and log into your OCP cluster, substituting the --token and --server parameters with your values:

        oc login --token=<token> --server=<server>

      If you are unsure of these values, click your user ID in the OpenShift web console and select "Copy Login Command".

    2. Create a fork of the IBM GitHub Enterprise repository containing the IBM API Connect configuration scripts that the OpenShift pipeline we created in the steps above will execute. Go to https://github.ibm.com/techaeta/apic-config-pipeline and fork the repository to you personal IBM GitHub Enterprise organization.

      Info

      The scripts that will carry out the actions against our IBM API Connect instance that will get executed by and within the IBM API Connect configuration pipeline have been hosted on IBM Enterprise GitHub so that it reflects more closely any client situation where all the code will be sitting behind a firewall on restricted source code repositories.

    3. Create a GitHub Personal Access Token so that the IBM API Connect configuration OpenShift pipeline has permissions to access your IBM GitHub Enterprise account. Provide just repo and admin:repo_hook permissions.

    4. Once we have the above, create the following secret containing your IBM GitHub Enterprise Username and Personal Access Token

      oc create secret generic apic-pipeline-git -n ci \
--GIT_USERNAME=<your_ibm_github_enterprise_username> \
--GIT_PRIV_TOKEN=<your_ibm_github_enterprise_personal_access_token>

    5. Create the following secret that will hold your Email Server configuration for notifications:

      oc create secret generic apic-config-email-server -n ci \
--EMAIL_HOST=<your_email_server_host> \
--EMAIL_PORT=<your_email_server_port> \
--EMAIL_USERNAME=<your_email_server_username> \
--EMAIL_PASSWORD=<your_email_server_password>

      You can read more about configuring an email server for notifications in the IBM API Connect documentation here.

      You can create your own dummy email server at https://mailtrap.io/.

    6. Create the following secret that will hold your Provider Organization configuration:

      oc create secret generic apic-pipeline-provider-org -n ci \
--PROV_ORG_OWNER_USERNAME=<provider_organization_owner_username> \
--PROV_ORG_OWNER_PASSWORD=<provider_organization_owner_password> \
--PROV_ORG_OWNER_EMAIL=<provider_organization_owner_email> \
--PROV_ORG_OWNER_FIRST_NAME=<provider_organization_owner_first_name> \
--PROV_ORG_OWNER_LAST_NAME=<provider_organization_owner_last_name> \
--PROV_ORG_NAME=<provider_organization_name> \
--PROV_ORG_TITLE=<provider_organization_title> \
--PROV_ORG_CATALOG_NAME=sandbox \
--PROV_ORG_REALM=provider/default-idp-2

      You can read more about creating a Provider Organization in the IBM API Connect documentation here.

    We are now ready to execute the OpenShift pipeline that will configure our IBM API Connect instance we got deployed in the previous chapter of this tutorial.

    1. Go again to Pipelines --> Pipelines on the left hand side menu of your OpenShift web console and click on the apic-post-install-config pipeline. Then, click on Start on the Actions drop down menu at the top right corner.

    2. Make sure that the values for the pipeline parameters on the Start Pipeline configuration menu that pops up are correct. Specially the value for the git-apic-pipeline-git-url pipeline parameter which should be the value for the IBM GitHub Enterprise repository containing the IBM API Connect configuration scripts that the OpenShift pipeline will execute and that we forked in a previous step within this section.

    3. Click Start. This will create a Pipeline Run object that represents this specific execution of the pipeline.

    You can follow along the execution of this OpenShift Pipeline Run by clicking the Logs tab at the top bar. If this execution of the IBM API Connect configuration pipeline finishes successfully you should see and END message at the bottom of the logs as well as a green check mark on the task the Pipeline Run has executed on the left hand side.

    If you switch back to the Details tab of the Pipeline Run, you should also see green check marks both at the Pipeline Run name at the top and below it in the task that has been executed by this Pipeline Run that is displayed on the Pipeline Run Details section.

    And in the main Pipelines section of your OpenShift web console you should also see the last Pipeline Run for the IBM API Connect config Pipeline in green

    Finally, if you open your IBM API Connect Cloud Manager again, you will see that the IBM API Connect config pipeline has configured a Topology for you with the default IBM API Connect subsystems that get installed so that we now have a working IBM API Connect environment.

    And in the IBM API Connect API Manager, we can see that the default Gateway service that has been configured in the IBM API Connect Cloud Manager Topology has been registered with the default Sandbox catalog, that gets created for the Provider Organization we specified in the IBM API Connect configuration pipeline, so that we can securely publish, expose and access our application APIs.

    Congratulations!

    You have successfully created and executed the IBM API Connect configuration pipeline that configures all of the IBM API Connect subsystems that the IBM API Connect Operator creates for your IBM API Connect instance so that you now have your API Connect instance ready to start working with.
