    IBM API Connect Multi-cloud Deployment¶

    Abstract

    This document describes the steps to carry out a multi-cloud deployment of IBM API Connect across different cloud providers, IBM Cloud and Amazon AWS, all using RedHat OpenSift Container Platform as the underlying unified platform.

    Overview¶

    As introduced in the create multi-cloud clusters section for the IBM API Connect Multi-Cloud use case, we are deploying IBM API Connect subsystems separately in different Red Hat OpenShift clusters. We will, at least, have 2 clusters. One, considered the control plane or brain cluster, will host the IBM API Connect Management and Portal components. The other, or others, will host the IBM API Connect Gateway and Analytics components.

    Create the IBM API Connect Multi-cloud profile¶

    1. Locate your GitOps repository

      If necessary, change to the root of your GitOps repository, which is typically $HOME/git.

      Issue the following command to change to your GitOps repository:

      cd $HOME/git
cd multi-tenancy-gitops

    2. Create the apic-multi-cluster profile. This will create the apic-multi-cluster profile under the 0-bootstrap/others directory which is where all the multi-cluster profiles are located.

      cd doc/scenarios/apic-multi-cluster/scripts
NAME=<your-gateway-and-analytics-cluster-name> ./create-apic-multi-cluster-profile.sh

      where <your-gateway-and-analytics-cluster-name> is a unique identifier for this first IBM API Connect Gateway and Analytics cluster. We must provide a unique identifiers since you could create as many IBM API Connect Gateway and Analytics clusters as needed.

      You should see the following output:

      $ NAME=test ./create-apic-multi-cluster-profile.sh

Creating an IBM API Connect multi-cluster profile under 0-bootstrap/others
The IBM Gateway and Analytics cluster will be called test
Done
You can find the new apic-multi-cluster profile under 0-bootsrap/others/apic-multi-cluster

    3. Check the apic-multi-cluster profile structure

      cd $HOME/git
cd multi-tenancy-gitops
tree 0-bootstrap/others/apic-multi-cluster -L 2

      You should see the following output:

      0-bootstrap/others/apic-multi-cluster
├── bootstrap-management-portal-cluster.yaml
├── bootstrap-test-gateway-analytics-cluster.yaml
├── management-portal-cluster
│   ├── 1-infra
│   ├── 2-services
│   ├── 3-apps
│   ├── kustomization.yaml
│   └── scripts
└── test-gateway-analytics-cluster
    ├── 1-infra
    ├── 2-services
    ├── 3-apps
    ├── kustomization.yaml
    └── scripts
      (*) Bear in mind that we have used test for the name of the IBM API Connect Gateway and Analytics cluster.

    We can see we have the structure for the two clusters we are going to install IBM API Connect subsystems on. The IBM API Connect Management and Portal cluster will be described and GitOps driven from the management-portal-cluster folder and the bootstrap ArgoCD application will be created from the bootstrap-management-portal-cluster.yaml file. The IBM API Connect Gateway and Analytics cluster will be described and GitOps driven from the test-gateway-analytics-cluster folder and the bootstrap ArgoCD application will be created from the bootstrap-test-management-portal-cluster.yaml file (remember that we will have one folder and bootstrap yaml file per IBM API Connect Gateway and Analytics cluster). Also, once again, bear in mind we have chosen test as the unique identifier for this particular IBM API Connect Gateway and Analytics cluster.

    Customize the apic-multi-cluster profile¶

    Since we have created a new profile under 0-bootstrap/others for our IBM API Connect Multi-Cloud use case, we need to customize this profile for our clusters as well. The ArgoCD Application and AppProject resources and kustomization.yaml resources contains environment variables which will need to be updated to your specific GitHub organization and branch.

    1. Run the customization script

      Let's customize the cloned multi-tenancy-gitops repository with the relevant values for our cluster.

      The sample GitOps repository provides the set-git-source.sh script to make this task easy.

      Run script to replace the git url and branch to your git organization where you created the git repositories:

      GIT_ORG=<Your Organization> GIT_BRANCH=master ./scripts/set-git-source.sh

      The script will list customizations it will use and all the files that it customizes:

      $ GIT_ORG=prod-ref-guide  GIT_BRANCH=master ./scripts/set-git-source.sh
Setting kustomization patches to https://github.com/prod-ref-guide/multi-tenancy-gitops.git on branch master
Setting kustomization patches to https://github.com/prod-ref-guide/multi-tenancy-gitops-infra.git on branch master
Setting kustomization patches to https://github.com/prod-ref-guide/multi-tenancy-gitops-services.git on branch master
Setting kustomization patches to https://github.com/prod-ref-guide/multi-tenancy-gitops-apps.git on branch master
done replacing variables in kustomization.yaml files
git commit and push changes now

      You can examine your local clone of the GitOps repository to verify these customizations. You should see lots of amended ArgoCD YAMLs that point to your GitOps repository on GitHub.

    2. Add all changes in the current folder to a git index, commit them, and push them to GitHub:

      git add .
git commit -s -m "Created apic-multi-cluster profile and customized"
git push origin $GIT_BRANCH

    Deploy the IBM API Connect Management and Portal components¶

    We have now the apic-multi-cloud profile apparently set up and ready to get deployed. However, we still need to configure the last piece of the puzzle which isn't other than specifying what is the domain and a block storage class name of the Red Hat OpenShift cluster where the IBM API Connect Management and Portal components will get deployed to.

    To find out what the domain of the the Red Hat OpenShift cluster where the IBM API Connect Management and Portal components will get deployed to,

    1. Ensure you are logged in to the cluster

      Start a terminal window and log into your OCP cluster, substituting the --token and --server parameters with your values:

      oc login --token=<token> --server=<server>

      If you are unsure of these values, click your user ID in the OpenShift web console and select "Copy Login Command".

      Warning

      Make sure you are logging into the IBM API Connect Management and Portal cluster

    2. Get the OpenShift domain from the OpenShift web console route

      oc get routes console -n openshift-console

      You should see an output similar to the following:

      NAME      HOST/PORT                                                                             PATH   SERVICES   PORT    TERMINATION          WILDCARD
console   console-openshift-console.itzroks-270005wyhs-xxxxx.eu-gb.containers.appdomain.cloud          console    https   reencrypt/Redirect   None

      where the OpenShift domain will be the domain after console-openshift-console of the HOST/PORT value of the route. In our case, it would be itzroks-270005wyhs-xxxxx.eu-gb.containers.appdomain.cloud

    Find out a block storage class available in your OpenShift cluster for the storage needed for the IBM API Connect Management and Portal components. You can check the available storage classes on your OpenShift cluster by executing:

    oc get storageclass

    Once you have your OpenShift cluster domain and a block storage class, we need to provide these to the IBM API Connect Management and Portal component instances that are defined in the management-portal-cluster folder that is driving the GitOps for our IBM API Connect Management and Portal cluster. More precisely, you can see these instances under apic-multi-cluster/management-portal-cluster/2-services/argocd/instances/ibm-apic-management-portal-instance/instances. You can change things manually or you can use the following script we provide to make the appropriate changes.

    1. Go to the scripts directory

      cd 0-bootstrap/others/apic-multi-cluster/management-portal-cluster/scripts

    2. Execute the set-storage-ocp-domain.sh script

      STORAGE=ibmc-block-gold OCP_DOMAIN=itzroks-270005wyhs-xxxxx.eu-gb.containers.appdomain.cloud ./set-storage-ocp-domain.sh

    3. Add all changes in the current folder to a git index, commit them, and push them to GitHub:

      git add .
git commit -s -m "Setting the OCP domain and block storage class"
git push origin $GIT_BRANCH

    4. Go back to the apic-multi-cluster profile folder

      cd ../../

    Next, we must have an IBM Entitlement Key to be able to pull IBM software down from IBM's software registry available in the Red Hat OpenShift project where we are deploying the IBM API Connect Management and Portal components. That namespace is tools.

    1. Create the tools namespace

      oc new-project tools

    2. Log in to MyIBM Container Software Library with an IBMid and password associated with the entitled software.

    3. Select the View library option to verify your entitlement(s).
    4. Select the Get entitlement key to retrieve the key.

    5. Create a secret containing your IBM Entitlement Key in the tools namespace. 

      oc create secret docker-registry ibm-entitlement-key -n tools \
--docker-username=cp \
--docker-password="<entitlement_key>" \
--docker-server=cp.icr.io

    6. Link that docker-registry secret containing your IBM Entitlement Key with the default secret for pulling Docker images within your Red Hat OpenShift project

      oc secrets link default ibm-entitlement-key --for=pull

    Finally, we are all set to bootstrap the IBM API Connect Management and Portal cluster with the initial ArgoCD application that will connect the ArgoCD instance that is running in the cluster that we deployed in the Configure the cluster section and this GitOps repository where we have defined, as code, what we want to deploy in the cluster.

    All you need to do is to apply the bootstrap-management-portal-cluster.yaml file under the 0-bootstrap/others/apic-multi-cluster folder.

    oc apply -f bootstrap-management-portal-cluster.yaml

    If you go to you ArgoCD web console, you would now see ArgoCD applications being created. These applications will be the responsible for watching over the different code in the GitOps repo and make sure that what is specified there gets created in the OpenShift cluster.

    If you go to your OpenShift web console under the Operators --> Installed Operators section on the left hand side of the menu, you should be able to see how the different operators that we have specified in the GitOps repository to be deployed are starting to get deployed.

    Tip

    While the IBM API Connect Management and Portal components are getting deployed in this cluster, you can jump to the next section to start deploying the IBM API Connect Gateway and Portal components in their respective cluster

    After around 45 minutes, you can check that the ArgoCD web console now displays all the ArgoCD applications in green

    the OpenShift web console shows that all of the operators have been successfully installed

    and, finally, if you go under the tools namespace for the Operators --> Installed Operators, click on the IBM API Connect operator and click on the All instances tab, you should see that your management and portal are Running

    Deploy the IBM API Connect Gateway and Analytics components¶

    You have successfully installed the IBM API Connect Management and Portal components and the IBM API Connect Gateway and Analytics components in their respective clusters achieving an IBM API Connect Multi-cluster scenario. You can now jump to the Configure API section
